2 Grundlagen

2.1 Anforderungen innerhalb der TU Dortmund und daraus resultierende Erweiterungen von PAIA

Die UB Dortmund benötigt einen nach aussen vom Bibliothekssystem (ILS) unabhängigen Service für Kontofunktionen aus Nutzersicht. Zu diesen zählen:

  • Abrufen von Informationen zu ausgeliehenen Medien
  • Abrufen von Informationen zu bestellten bzw. vorgemerkten Medien
  • Anzeige von offenen Gebühren
  • Aufgeben und Stornieren von Bestellungen bzw Vormerkungen
  • Verlängerung von Ausleihfristen

Die Notwendigkeit für einen solchen Service ergibt sich aus einer Vielzahl von neuen vernetzten Dienstleistungen innerhalb der TU Dortmund.

Zentrales Szenario ist die Einbindung der Kontofunktion für Nutzende in das neue Recherchesystem der UB Dortmund Katalog plus. In Kombination mit der Frage nach der Ermittlung von Live-Informationen zur Verfügbarkeit von Medien in der UB Dortmund, ergab sich die Möglichkeit mit DAIA und PAIA diese Aufgabe zu erfüllen.

Für einige spezielle Funktionen wurde die originale PAIA-Spezifikation erweitert.

Die Authentifizierungs- und Autorisierungsfunktionen werden mittels OAuth 2.0 realisiert. Eine Implementierung über das ILS steht mit dem OAuth2Service.

4 Modul PAIA Core

PAIA Core ist ein Ressource Server im Sinne von OAuth 2.0. Der Zugriff auf diese Ressourcen erfolgt nach dem OAuth 2.0-Konzept.

4.1 Funktion patron

Ziel
Diese Funktion ermöglicht es, Informationen aus einem Benutzerkonto zu ermitteln.

HTTP verb and URL
GET https://api.ub.tu-dortmund.de/paia/core/{account}

Scope
read_patron

Felder "Response"

  • name = vollständiger Name
  • email = E-Mail-Adresse
  • usergroup = Benutzergruppe
  • faculty = Fakultät
  • expires = Ablaufdatum der aktuellen Nutzungsfrist
  • status = aktueller Kontostatus

Kontostatus:

  • 0 = aktiv und keine Sperre
  • 1 = nicht aktiv
  • 2 = nicht aktiv, weil Gültigkeitszeitraum abgelaufen (enthält auch SunRise-Sperre 99)
  • 3 = nicht aktiv: bitte im Ausleihzentrum der UB nachfragen

Beispiel:

GET /paia/core/ubtest05/patron HTTP/1.1
Host: api.ub.tu-dortmund.de
Accept: application/json
Authorization: Bearer a0dedc54bbfae4b

HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
X-Accepted-OAuth-Scopes: read_patron
X-OAuth-Scopes: read_patron, read_fees, read_items, write_items

{
  "name": "UB-TEST-05, UB-TEST-05", 
  "email": "jane@example.org",
  "usergroup": "25",
  "faculty": "4",
  "expires": "2015-05-18",
  "status": 0
}

4.2 Funktion fullpatron

Ziel
Diese Funktion ermöglicht es, detailierte Informationen aus einem Benutzerkonto zu ermitteln und dient der Überprüfung nach der Durchführung administrativer Funktionen via PaaaService. Diese Funktion ist nur verfügbar, falls für den Anfragenden der Scope write_patron freigeschaltet ist.

HTTP verb and URL
GET https://api.ub.tu-dortmund.de/paia/core/{account}/fullpatron

Scope
write_patron

Felder "Response"

  • name = vollständiger Name
  • email = E-Mail-Adresse
  • expires = Ablaufdatum der aktuellen Nutzungsfrist
  • status = aktueller Kontostatus
  • gender = Geschlecht (1 = männlich, 2 = weiblich)
  • dateofbirth = Geburtsdatum (yyyy-mm-dd)
  • city = Stadt
  • postalcode = Postleitzahl
  • street = Straße
  • usergroup = Benutzergruppe (25 = TU-Mitarbeiter, 30 = TU-Studierende, 35 = UA Ruhr-Studierende)
  • externalid = Matrikelnummer oder Personalnummer
  • faculty = Fakultät

Kontostatus:

  • 0 = aktiv und keine Sperre
  • 1 = nicht aktiv
  • 2 = nicht aktiv, weil Gültigkeitszeitraum abgelaufen (enthält auch SunRise-Sperre 99 = "zur Löschung ")
  • 3 = nicht aktiv: bitte im Ausleihzentrum der UB nachfragen

Beispiel:

GET /paia/core/19026118/fullpatron HTTP/1.1
Host: api.ub.tu-dortmund.de
Accept: application/json
Authorization: Bearer a0dedc54bbfae4b

HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
X-Accepted-OAuth-Scopes: read_patron
X-OAuth-Scopes: write_patron

{
   "name": "Bond, James",
   "email": "agent@queen.uk.co",
   "expires": "2015-03-12",
   "status": 2,
   "gender": "1",
   "dateofbirth": "1946-08-12",
   "city": "London",
   "postalcode": "54321",
   "street": "Royal Street 1",
   "usergroup": "35",
   "externalid": "123456789X",
   "faculty": "4"
}

4.3 Funktion items

Ziel
Diese Funktion liefert alle ausgeliehenen, vorgemerkten und bestellten Titel in einem Benutzerkonto. Die erfolgreiche Antwort dieser Funktion ist um zwei Felder gegenüberder PAIA-Spezifikation erweitert:

  • recalled = boolean - gibt an, ob ein Medium bereits zurückgefordert wurde
  • renewable = boolean - gibt an, ob die Ausleihe eines Mediums generell verlängert werden kann

HTTP verb and URL
GET https://api.ub.tu-dortmund.de/paia/core/{account}/items

Scope
read_items

Felder "Response"

  • doc = Liste von Dokumenten

Beispiel:

GET /paia/core/19026118/items HTTP/1.1
Host: api.ub.tu-dortmund.de
Accept: application/json
Authorization: Bearer a0dedc54bbfae4b

HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
X-Accepted-OAuth-Scopes: read_items
X-OAuth-Scopes: read_patron, read_fees, read_items, write_items

4.8 Funktion newfee

deprecated: Wird in Release v1.0 durch newfee im PaaaService ersetzt!

Ziel
Diese Funktion ermöglicht es, Gebühren zu buchen. Für den Datentyp money siehe PAIA 2.5 Data types

Es kann mit einem Funktionsaufruf eine Gebührenbuchung übergeben werden. Im Fehlerfall wird definitiv keine Position gebucht.

HTTP verb and URL
POST https://api.ub.tu-dortmund.de/paia/core/{account}/newfee

Scope
write_fees

Felder

  • (*) account = Uni-Account via URL
  • fee = Fee
    • amount = Betrag. Für den Datentyp money siehe PAIA 2.5 Data types
    • about = Gebührengrund
    • reason = Code des Gebührengrundes (4 oder 04 für "Unicard-Ersatzgebühr")

(*) = Pflichtfeld

Beispiel:

POST /paaa/BondJamesBond/newfee
Host: api.ub.tu-dortmund.de
Accept: application/json
Authorization: Bearer a0dedc54bbfae4b

Content-Type: application/json
Content-Length: 108

{
    "amount": "3.00 EUR",
    "about": "Unicard-Ersatzgebühr",
    "reason": "4",
}

Response:

{"success":"true"}

Fehler:

  • errorcode = string Reason for error, code
  • errortext = string Reason for error, text

Codes:

  • 570 = unknown patron
  • 571 = patron blocked

Beispiele:

{"success":"false","errorcode":"570","errortext":"unknown patron"}
{"success":"false","errorcode":"571","errortext":"patron blocked"}
{"success":"true"}
